## 1. Goal of Phase 2

Take the system from prototype to a beta usable in real conditions:

- Automatic deploy triggers from Git commits (CI/CD).
- Component test framework (unit pre-deploy, integration/smoke
  post-deploy).
- Automatic rollback on failed tests/healthchecks.
- Hot deploy and blue-green for components that support them.
- Canary rollout strategy for host/component bindings.
- Encrypted credential vault with a pluggable backend.
- Complete MCP server, with all verbs documented in `architecture.md §3`.
- Rich web UI (React/Vite) with log streaming, metric charts, and
  deployment history.
- Mature skill with decision flows.

**Out of scope:** Kubernetes, advanced observability (Prometheus/tracing),
HA, mTLS, CLI (all in Phase 3).

## 2. Prerequisites

- Repository as it stands at the completion of Phase 1 (all acceptance
  tests passing).
- Node 20+ and pnpm or npm (for the React UI).
- A real Git repository available (even a local bare repo) for the
  git-sync tests.

### Group B — Credential vault

B1. `control-plane/app/credentials/vault.py`: interface with methods
    `get(ref: str) -> SecretValue`, `put`, `delete`, `list`.

B2. `control-plane/app/credentials/file_backend.py`: encrypted-file
    backend. Master key derived from a passphrase (scrypt with
    conservative parameters: N=2**15, r=8, p=1). Format: version header +
    salt + AES-256-GCM over the JSON.

B3. CLI helper `python -m app.credentials.cli` (or script in `scripts/`):
    - `vault init` — create an empty vault.
    - `vault set <path> <value>` — add/update a secret.
    - `vault get <path>` — print a secret (development only).
    - `vault list` — list paths.

B4. Orchestrator integration: when a deploy payload needs a secret
    (`{{ vault://... }}`), secrets are resolved just-in-time before being
    sent to the daemon and passed in the `secrets` field of the
    `request.deploy` message. Never written to disk in cleartext.

B5. Git credentials management: same vault, conventional path
    `git.<alias>`. The control-plane clone uses the resolved credentials.

**Unit tests (`tests/unit/test_credentials.py`):**
- Init + set + get + list + delete.
- Wrong passphrase → error (no crash).
- Corrupted file → error with a clear code.
- Secret referenced in a component template resolved correctly.

### Group C — Git-sync component

C1. New module `control-plane/app/gitsync/`:
    - `poller.py`: periodic polling of the tracked refs for all
      `source.type: git` components.
    - `webhook.py`: endpoint `POST /api/webhooks/{provider}` that accepts
      GitHub/GitLab/Gitea/Bitbucket payloads, validates the signature,
      and notifies the sync.
    - `sync.py`: central logic. When it detects a new commit:
      - Updates the "drift detected" state for the component.
      - Consults the policy (`auto_deploy: true|false`, default false).
      - If auto_deploy, triggers the deployment via the engine.

C2. Storage of tracked commits (last commit seen for
    `<component_id,branch>`) in a SQLite table.

C3. Configuration in `deployment.yaml`:
    ```yaml
    components:
      api:
        source:
          type: git
          repo: ...
          ref: main
          sync:
            poll_interval: 5m         # default
            auto_deploy: true
    ```

C4. UI: "drift detected" badge on components with a new commit not yet
    deployed. "Deploy latest" button.

**Tests:**
- `test_gitsync_poller.py`: mocked repo; verifies detection of the new
  commit.
- `test_webhook.py`: signed and unsigned GitHub payload; verifies
  processing or rejection.
- e2e `test_git_autodeploy.py`: local bare git repo → commit push →

### Group D — Component test framework

D1. Extend `request.tests.run` in the protocol: the payload specifies
    which tests to run (`unit`, `integration`, `smoke`, `all`).

D2. In the daemon, a handler that:
    - For `command` tests: runs in the component's working_dir,
      captures output, returns the structure `{name, ok, duration_ms,
      stdout_tail, stderr_tail, exit_code}`.
    - For `http` tests: sends the request and verifies status/body.

D3. Orchestrator integration:
    - `pre_deploy` with `blocking: true` → on failure, deploy cancelled
      and an error is returned.
    - `post_deploy` with `blocking: true` → on failure, automatic

    - Results persisted in the deployment history.

D4. UI: "Tests" view per component with the history of results.

**Tests:**
- `test_tests_runner_daemon_test.go`: command-test execution with
  success and failure; output capture; timeout.
- `test_orchestrator_with_tests.py`: deploy with a failed pre test →
  deploy cancelled. Deploy with a failed post test → rollback called.

### Group E — Hot deploy, blue-green, canary

E1. Daemon: for `deploy_mode: hot`, extend the runners:
    - `systemd`: run `systemctl reload <unit>` if the component has an
      `ExecReload` defined; otherwise load the new binary version
      side-by-side and run `systemctl daemon-reload && systemctl restart`
      with a strategy that preserves uptime as much as possible.
    - `docker`: start a new container with a temporary name, positive
      healthcheck, swap the `container_name` (rename the old one, rename
      the new one), remove the old one. Published ports: require either
      a reverse proxy in front or that the component tolerates a few
      seconds of overlap (`--publish-all` option during the swap
      window).

E2. Daemon: for `deploy_mode: blue_green`, explicit variant that keeps
    two permanent installations and switches "traffic" (which, in the
    absence of a load balancer, means: which container exposes the
    declared ports).

E3. Orchestrator: for the `canary` strategy:
    - Computes the "instances" (in the single-host case, ≥1 host virtual
      group; in the multi-host case, applies to the group).
    - Deploys `initial_fraction` → waits `verify_duration` + healthcheck.
    - Expands by `step_fraction` up to 100% or rolls back if verification
      fails.

E4. New field `reload_triggers`: when a deployment changes only a `hot`
    component's config, perform a reload instead of a restart.

**Tests:**
- `test_hot_deploy_docker_test.go` (daemon): hot deploy of nginx with a
  different config; verifies zero-downtime (continuous `curl` requests
  every 100ms during the swap → no 5xx).
- `test_canary_orchestrator.py`: two simulated hosts, verifies gradual
  rollout and interruption on failure.

### Group F — Automatic rollback

F1. Daemon: keep a history of the last N (default 5) deployed hashes per
    component, with the artifacts needed to go back (tagged docker
    image, tarball of the previous code, previous config).

F2. `request.rollback` payload: `{component_id, steps_back: 1}` or
    `{component_id, to_hash: "..."}`.

F3. Orchestrator: on failure of a post-deploy `healthcheck` after N
    attempts, or failure of a blocking post-deploy test, automatically
    issues `request.rollback` for the offending component and, if the
    strategy requires it, for the components deployed in the same
    session.

F4. API: `POST /api/components/{id}/rollback?steps=1`.

F5. UI: "Rollback" button in the component panel with a "how many steps
    back" selector.

**Tests:**
- `test_rollback_daemon_test.go`: deploy A → deploy B → rollback → state
  returns to an identical A.
- `test_auto_rollback.py`: deploy with a deliberately failing
  healthcheck → orchestrator issues an automatic rollback within the
  timeout.

### Group J — Granular lifecycle primitives

Phase 1 allows applying a whole `deployment.yaml` but lacks primitives
for (a) deploying a component with a package the agent already has
locally (e.g. a binary built on its own machine) without going through
Git; (b) removing a component idempotently — today `to_remove` in the
diff is skipped by design; (c) inspecting a host's state in a single
round-trip. Phase 2 fills these three gaps.

#### J1 — Artifact upload + `source.type: artifact`

CP-side:
- REST endpoints:
  - `POST /api/artifacts` (multipart or JSON with `content_b64`): upload
    of a file, returns `{artifact_id, sha256, size, created_at}`.
  - `GET /api/artifacts` list, `GET /api/artifacts/{id}` metadata,
    `DELETE /api/artifacts/{id}`.
- Local storage with deduplication by sha256 and configurable TTL
  (default 24h); metadata in the DB.
- When the orchestrator needs to deploy a component with
  `source.type: artifact`, it includes the bytes in the `request.deploy`
  payload to the daemon (reusing the existing `inline_tarball` subtype
  for archives, or a new `binary_executable` for single executables).

YAML schema (see `yaml-schema.md`):
```yaml
components:
  api:
    source:
      type: artifact
      artifact_id: "a-7f3a..."   # uploaded via upload_artifact
    run: { ... }
```

MCP / API:
- Tool `upload_artifact(path, name?)` → `{artifact_id}`. The agent
  provides the path of a file on its machine (or base64 bytes); the CP
  registers it.
- Tool `update_component(component_id, source?)` to update **a single**
  component without modifying the canonical YAML:
  - If `source` is provided, temporarily replaces the source and
    deploys (useful for "try this artifact without committing the
    change").
  - If omitted, reuses the current source (equivalent to `deploy
    --component=X`).
- Analogous REST API: `POST /api/components/{id}/update` with optional
  body `{source: {...}}`.

**Special case — daemon self-update** (does not require a dedicated
verb: reuses J1):
- The daemon is declared as a managed component in the host's YAML:
  ```yaml
  components:
    maestrod:
      source: { type: artifact, artifact_id: "..." }
      run:
        type: systemd
        unit_name: maestro-daemon
        command: /usr/local/bin/maestrod --config /etc/maestrod/config.yaml
      deploy_mode: blue_green
      self_update: true
  ```
- With `self_update: true`, the daemon applies the "replace-on-exit"
  pattern: writes the new binary to `/usr/local/bin/maestrod.new`,
  spawns a child process on an alternate endpoint for the healthcheck,
  performs an atomic swap of the binary + `systemctl restart
  maestro-daemon`. Safety: the new process must reconnect to the CP
  within the timeout (default 60s), otherwise an automatic rollback
  restores the previous binary.

Tests:
- `test_artifact_upload.py` (unit): upload + sha256 dedup + TTL + delete.
- `test_artifact_deploy_test.go` (integration): deploy of a tarball
  uploaded via artifact, end-to-end.
- `test_self_update.py` (e2e): build `maestrod` locally with
  `Version = "0.2.0-test"`, upload, deploy as self-update, verify
  version bump after reconnection and downtime ≤ 10s.

#### J2 — Component removal

Daemon-side:
- Handler `request.component.remove` with payload
  `{component_id, keep_volumes?: bool, keep_state?: bool}`.
- Runner-specific execution:
  - Docker: `docker rm -f`, remove declared volumes (skip if
    `keep_volumes`).
  - Systemd: `systemctl stop + disable`, remove the unit file, remove
    `/opt/maestro/<id>/`.
- Delete the row from the state store (unless `keep_state` for audit).
- Emits `event.component_removed`.

Orchestrator:
- Implement the `to_remove` branch in the diff (today skipped with a
  `Phase 1: we don't remove automatically` comment).
- `apply_config?prune=true` actually performs removals; default
  `prune=false` preserves the current behavior.
- MCP tool `remove_component(component_id, keep_volumes?)` for targeted
  removal without touching the YAML.

Tests:
- `test_remove_daemon_test.go`: deploy → remove → verify that
  container/unit are gone, correct volumes behavior with/without
  `keep_volumes`.
- `test_orchestrator_prune.py`: apply with `prune=true` removes
  components no longer in the YAML.

#### J3 — Host diagnostics

Daemon handler `request.host.diagnostics` returns an aggregated snapshot
(token optimization: the agent gets everything in a single round-trip
instead of calling N times):

```json
{
  "host_id": "host1",
  "os": { "name": "Ubuntu", "version": "24.04", "kernel": "6.8..." },
  "uptime_sec": 86400,
  "cpu": { "count": 4, "load_1m": 0.4, "load_5m": 0.3 },
  "memory_mb": { "total": 8192, "used": 2048, "available": 6144 },
  "disk": [ { "path": "/", "total_gb": 100, "used_gb": 43 } ],
  "runtimes": {
    "docker":  { "active": true, "version": "29.1.3" },
    "systemd": { "active": true, "version": "255" }
  },
  "daemon": { "version": "0.2.0", "uptime_sec": 3600, "reconnects": 0 }
}
```

MCP tool `get_host_diagnostics(host_id)`.

Tests:
- `test_diagnostics_daemon_test.go`: mocked system commands, verify
  output structure.
- `test_mcp_diagnostics.py`: MCP round-trip, payload < 2 KB per host
  with ≤ 5 components.

## 5. Phase 2 acceptance suite

### Acceptance 1 — Phase 1 regression

The entire Phase 1 acceptance suite must continue to pass without

### Acceptance 2 — Credential vault

- `vault init` + `vault set db/password "secret"` + deploy of a component
  that references `{{ vault://db/password }}` → component receives the
  correct env var.
- Stop the control plane, restart with the correct passphrase → vault
  reopenable.
- Restart with a wrong passphrase → clear error, no crash.

### Acceptance 3 — Git-sync

- Configure a component with `sync.auto_deploy: true`.
- Push a commit on the tracked branch.
- Within the poll interval, or immediately if via webhook: the component
  is redeployed at the new commit. Audit log retains the event.

### Acceptance 4 — Test framework

- Deploy with `tests.unit.blocking: true` that fails (`command: false`)
  → deploy cancelled, error reported.
- Deploy with `tests.smoke` that fails post-deploy → automatic rollback
  executed, final state = previous.

### Acceptance 5 — Hot deploy

- Hot deploy of nginx → while it is happening, a `curl` loop against
  the published port produces no errors (tolerate at most 2 errors out
  of 100 requests).

### Acceptance 6 — Canary

- Canary deploy across 3 (simulated) hosts with `initial_fraction: 0.34`,
  `step_fraction: 0.34`. The rollout progresses to 1/3, then 2/3, then
  3/3 after positive healthchecks.
- Introduce a failure mid-rollout → rollout interrupted and already
  deployed hosts rolled back.

### Acceptance 7 — Complete MCP

A Python script uses the MCP client and invokes *every* documented verb;
each call returns an object with the expected structure.

### Acceptance 8 — Token usage

- `get_deployment_history` returns only the last N deploys (N

- API/MCP errors *always* have a `code` field (enum) + `message` +
  (when applicable) `suggested_fix`.
- `deploy` payload to the daemon stays ≤ 4 MB or fails with a clear
  error suggesting the use of `source.type: git` on the daemon side.

### Acceptance 9 — UI

Playwright e2e test: the complete user flow (no-auth login → edit YAML
→ apply → see dashboard populate → open logs → force drift → see the

### Acceptance 11 — Artifact upload + granular update

- The agent runs `upload_artifact` with a local tarball → receives an
  `artifact_id`.
- `update_component` on an existing component with
  `source: {type: artifact, artifact_id: "..."}` → the component is

### Acceptance 12 — Daemon self-update

  `/api/hosts`, the disconnection observed by the CP lasts ≤ 10 s, no

- Fault injection: if the new binary does not reconnect within 60 s,
  the daemon automatically reverts to `0.1.0` (rollback).

### Acceptance 14 — Diagnostics

## 7. Things the agent must **not** do in Phase 2

- Do not introduce Kubernetes.
- Do not introduce integrated Prometheus/tracing (basic metrics already
  in place continue to work).
- Do not introduce control-plane HA: single node is accepted.
- Do not introduce multi-user RBAC: UI stays accessible without auth or
  with a single admin token.
- Do not migrate to PostgreSQL: SQLite stays adequate in Phase 2.

## 8. Updated quality criteria

- Test coverage ≥ 85% Python, ≥ 75% Go.
- Rollback time ≤ 15 seconds per Docker component.
- Daemon reconnection time ≤ 10 seconds.
- No secrets written in cleartext to disk (verified by a dedicated test
  that searches for known patterns after a deploy).
